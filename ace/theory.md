# ACE — Atomic Constraint Theory
## A one-primitive model for project controls, spatial reasoning, and organizational memory

**Author.** Working document. Draft.
**Status.** Prototype implemented (~2,500-line single-file demo). Theory in review.
**Audience.** Expert reviewer, adversarial. Please attack the weakest claim first.

---

## 0. Abstract

Every project-controls system I have seen — Primavera P6, Microsoft Project, IBM Maximo, SAP PM, Asana, Jira, Procore, Aconex, Autodesk Construction Cloud — is built from many primitives: a task table, a resource table, a constraint table, an RFI table, a risk register, a materials system, a documents system, a change-order register. Each primitive requires its own database, its own vendor, its own permissions model, its own integration bill, and its own version of the truth. The industry has stopped noticing how much time is spent reconciling these systems and treating the reconciliation as an emergent, unavoidable cost of "digital transformation."

I claim there is only one primitive. I call it an *atom*. Every task, every constraint, every material delivery, every RFI, every risk, every drawing revision, every payment, every safety hazard, every stakeholder relationship, every verbal commitment — every noun a construction organization uses — is an atom. Every atom has the same seven fields and participates in the same two link types. State is computed by one fixed-point operation, which I call *derive*. Everything else — the Gantt, the WBS, the risk register, the model, the audit trail, the earned-value curve — is a view.

The prototype demonstrates that this is not merely a philosophical position: the same graph runs planning, forecasting, spatial layout, procedural documentation, morning triage, and audit narrative from a single ~50-line engine.

The obvious pushback is that some things are not atoms. § 12 addresses this directly by trying, for eight successive rounds of increasing difficulty, to find a counter-example. Each round fails. The paper takes seriously the possibility that a round nine exists.

## 1. Motivation — the five-tool problem

The typical field engineer on a nuclear refurbishment project logs into five systems before their first coffee. The scheduler tells them the pour is on for Wednesday. The RFI log says the design clarification is outstanding. The materials system says the rebar arrived Tuesday but does not know that the design clarification affects splice length. The risk register was updated last quarter. The morning stand-up read out a colored heatmap generated by a sixth system that reconciles the first five overnight.

The engineer does not think of this as five systems. They think of it as *the plan*, which is unreliable. When they act, they act against a mental model that has already synthesized the five sources into a single picture — a task that has a predecessor, a material state, an open question, and a risk exposure, all bound to one physical piece of work in one location. That synthesis is what management pays for. The five systems are the overhead.

The five-system architecture is a historical accident. Each vendor solved one part of the problem in isolation and each solution accreted a data model matched to that part. The integration industry then grew to bridge them. There is no principled reason planning and constraints should live in different tables. The engineer already treats them as the same thing.

The alternative direction is: pick a data model general enough that all five tools reduce to views on it. Prove that model is complete (or find the counter-example). Ship it as one file.

## 2. The atom

An **atom** is a record with seven fields:

    {
      id:       string          // globally unique
      name:     string          // human-readable
      type:     string          // free label — see § 11
      kind:     leaf | gate | rollup | verified | trivial   // § 5
      tags:     string[]        // free labels
      requires: id[]            // predecessors (§ 3.2)
      contains: id[]            // children (§ 3.1)
    }

Optional fields carry domain-specific data — `duration`, `cost`, `evidence`, `narrative`, `geometry`. These are values inside an atom, not separate entities.

The atom is not a class hierarchy. It is a single flat record. There are no subclasses. There is no "Task extends Atom." All differentiation lives in the `type` field, which is a string. The engine does not switch behavior on `type`; consumers do.

## 3. Links

There are two link types. I defend this claim in § 3.3.

### 3.1. Contains — the vertical link

`a.contains ∋ b` means atom `a` is the parent of atom `b`. Every atom has exactly one parent (or none, if it is the root). Contains is a tree.

The vertical link carries **rollup semantics**: an atom's completion depends recursively on its children's completion. Rollup is not a stored field; it is computed on read by § 4.

### 3.2. Requires — the horizontal link

`a.requires ∋ b` means atom `a` cannot proceed until atom `b` is complete. Requires is a *directed acyclic graph* (DAG). Cycles are rejected at insertion time.

The horizontal link carries **readiness semantics**: an atom is workable when everything it requires is complete. Readiness, like rollup, is derived.

### 3.3. Why no third link?

Every proposed third link type reduces to a tag plus one of the two link types:

- **"blocks"** is `requires` from the reverse angle.
- **"assigned-to"** is a `contains` link on the assignee's atom (the person owns the atom for now) plus a `role` tag.
- **"references"** is a tag: it carries no completion semantics.
- **"duplicates"** is either an alias (§ 11 · meta) or a rejection at insert time.
- **"caused-by"** is `requires` on a `narrative` atom that carries the cause.

The design rule is: if a proposed link type would carry no semantic that the existing two cannot, it is a tag; if it would carry a new completion semantic, it is a new atom type with a `requires` edge. Adding a third link type would make derivation ambiguous — we would have to decide which link's semantics dominate — and would multiply the graph's complexity without adding expressive power. I have not yet found a case that resists this reduction.

## 4. Derive — the fixed point

### 4.1. Semantics

An atom is *done* when:

- if `kind = manual`, a person has flipped it and provided any required narrative;
- if `kind = derived` (rollup or gate), every atom in `contains` is done AND every atom in `requires` is done.

Done is never stored. It is a function of the graph:

    done(a) = manualDone(a)                       if kind = manual
           = ∀ x ∈ a.contains ∪ a.requires : done(x)   otherwise

### 4.2. The algorithm

Because `contains` is a tree and `requires` is a DAG, the union is a DAG. A single topological pass suffices to resolve every atom's `done` value. The engine is:

    derive(atoms):
      cache ← ∅
      for each atom a in topological order:
        cache[a] ← done(a) using cache for children/requires
      return cache

Complexity is O(V + E). For a project with 10,000 atoms and average out-degree 3, this is one pass over ~40,000 edges, which is milliseconds in a browser.

### 4.3. Why derive, not store?

Storing `done` requires a maintenance path: on every mutation, walk the ancestors and update. Derivations that depend on `done` (readiness, critical path, forecast) then depend on the write path being correct. Bugs accumulate at the storage boundary — a stale `done` field is the single most common source of wrong dashboards in industrial project systems.

Deriving is bounded by the shape of the graph, not by the correctness of write paths. It is a proof, not a memory. This is the same discipline that persistent functional data structures use.

## 5. Five kinds

The `kind` field answers one question: *who or what declares the atom done?* Every completion I have found in construction, IT operations, regulatory work, and personal task management reduces to exactly one of five kinds. The kinds are not classes; they are semantic tags read by the derive step.

- **Leaf.** A person flips it, optionally with evidence and narrative. This is manual work.
- **Gate.** Pure derived. Waits on something else. RFIs and constraints live here. A gate has no inherent work — it is a check that fires when its predecessors clear.
- **Rollup.** Done when all its children are done. Areas, work packages. Carries no work of its own — it summarises children.
- **Verified.** Manual AND witness-signed. Regulatory or QA atoms need both the underlying work AND an authorized signatory. Two locks.
- **Trivial.** Born done. Historical facts, closed lessons, static references. Present so other atoms can `require` them; require no work.

### 5.1. Why exactly five?

The taxonomy of completion factors along two orthogonal axes:

- *Who declares?* — machine (derived from graph) or human (manual)
- *What is required?* — nothing (trivial), work only (leaf), children done (rollup), predecessors done (gate), work AND witness (verified)

Not every cell is populated. Machine × nothing is impossible (nothing to say). Machine × work is manual by definition. That leaves the five above. If you find a sixth, please write.

### 5.2. Reducibility

Every project system I have inspected can be encoded in these five kinds. A recurring task is a template atom (§ 11) that produces leaf instances. A milestone is a trivial atom at the end of a chain. A risk register entry is a gate atom whose `requires` list points at its triggering conditions. A commitment made in a hallway is a leaf atom with `narrative-required` and no `contains`.

## 6. Space — constraints as geometry

Positions are not stored. A position is an expression evaluated against a small vocabulary of named dimensions. The reactor building has a diameter `rb-d`; the turbine building's `x` coordinate is the expression `rb-d/2 + tb-w/2 + 15`, which evaluates to a number when the dimensions are supplied. Change `rb-d` from 46 m to 60 m and six atoms move.

This is the same primitive that carries the schedule. A building atom has `contains` (its sub-assemblies), `requires` (predecessor buildings whose services it depends on), and, in its otherwise-optional fields, a `geometry` expression. The engine reads this the same way it reads any atom.

The consequence is that the WBS and the site plan are two views of one graph. You cannot have a construction work area on the schedule that does not correspond to a footprint on the plan, because they are the same atom. Interference (§ 11 · clash) is a *negative* geometric constraint between two atoms — if they occupy the same envelope at the same time, the graph is over-determined and the engine emits a diagnostic.

## 7. Time

### 7.1. Duration is a triple

Each work atom carries `duration = [min, likely, max]`. A single number is a degenerate triple.

### 7.2. Critical path emerges

The critical path method is a walk over `requires`. Every atom's earliest finish is `max(requires' earliest finish) + duration.likely`. The longest chain from any root to a chosen end atom is the critical path. It has zero float. Nothing new.

The engine does not maintain a separate CPM data structure. CPM is a walk it runs on demand.

### 7.3. Monte Carlo

Sampling the duration triple with a triangular distribution and walking the graph yields a finish date. Repeating N times yields a histogram. Reading P50, P80, P90 off the histogram tells you what the plan actually promises. The seed determines the histogram exactly: for a given graph and seed, the P50 is deterministic.

The plan is not a date. The plan is a distribution.

### 7.4. Uncertainty as an atom

The three-value duration is a *representation* of uncertainty inside an atom. A more expressive representation is an `assumption` atom listed in `requires`: the assumption's own `manualDone` is `true` until challenged, at which point every atom that depends on it becomes not-ready. The engine now has a first-class handle on "the plan assumed a 5-day weather window" that goes red the day the forecast changes.

## 8. Narrative

### 8.1. Narrative is a tag

`narrative-required` is a tag on an atom. When such an atom is completed, the engine refuses to mark it done unless a written record is captured. This forces a human loop — no silent close on a regulator-facing atom.

### 8.2. Canned options are structure

Free-text narrative sounds noble and produces nothing. Real audit trails have a shape: was the outcome nominal, did rework happen, did a non-conformance open, was a witness present? The narrative modal offers *typed* choices per atom type, styled by severity, plus a free-text field for the residual. This is not a compromise. Structured narrative aggregates; free-text narrative does not. Both belong in the record. The choice is required; the free-text is optional; the audit trail becomes queryable.

### 8.3. Narrative accretes

Every state transition on a narrative-required atom is itself an atom of type `event` (§ 11 · meta). The audit trail is not a log — it is a subgraph. This lets you ask, of any atom, *what has this thing lived through?*

## 9. Literate procedures

### 9.1. Prose is graph

A standard operating procedure — the Word document on the wall of the trailer — reads as prose. In ACE, every named entity in the prose is a live atom pill: material, tool, RFI, permit, witness, PPE. The prose renders unchanged; the pills carry state derived from the shared graph. When the material's `manualDone` flips true, the pill turns green — inside the procedure.

### 9.2. The document runs

You do not maintain the procedure and the graph separately. The procedure IS the graph. Change a step, that's a graph mutation. Complete an atom, the prose lights up. There is no drift between what the crew reads and what the engine schedules, because there is one artifact.

### 9.3. Contrast with prevailing practice

In current practice, the procedure and the graph are dual-maintained. The procedure gets updated when audit season demands it; the schedule gets updated daily; the connection is a human who remembers to translate. Every organization I have seen this in produces a discrepancy within weeks. The discrepancy is invisible until an incident, when a lawyer discovers that the procedure the crew read differs from the sequence the schedule ran.

## 10. Flex — capacity vs sequence constraints

Earlier drafts of this paper claimed flex atoms have no `requires`. That was a bug in my thinking. Every atom is a constraint. Flex is no exception.

The distinction between flex work and critical-path work is not the *absence* of constraints. It is the *shape* of the constraint:

- **Sequence constraint.** `requires: REBAR` — points at a specific atom by id. Fires when that atom's `done = true`.
- **Capacity constraint.** `requires: any(crew where competency='civil' and status='idle')` — points at a *predicate over the atom set*. Fires when any atom matching the predicate exists.

Both are `requires` edges. The difference is whether the target is a name or a query. A capacity constraint's satisfaction depends on the *state of the population* rather than the state of a specific atom. The engine treats the two identically during derive.

This is why flex packages absorb variance. Their gates fire on availability, not on completion. When a crew becomes idle, the predicate matches, and the next flex atom becomes ready. The schedule does not need to model idle time explicitly; the graph makes it queryable.

The generalization is that `requires` targets should be first-class predicates. Names are a degenerate case of `id = 'REBAR'`. The uniform representation is cleaner, and it opens the door to §§ 13.1–13.2 below.

## 11. The taxonomy — 72 atom types

Every noun a construction organization uses is an atom. Here is what "every noun" means. Nine families, 72 types. Each type has the same seven fields; the semantics of the fields differ.

**1 · Work & structure.** project · area · cwp · iwp · task · step · milestone · flex.

**2 · Gates & controls.** constraint · rfi · risk · hazard · control · permit · exception · ncr · stop-work.

**3 · Verify & audit.** verify · test · audit · finding · corrective · narrative.

**4 · Resources.** material · consumable · tool · equipment · vehicle · ppe · inventory · lift-plan.

**5 · Humans.** person · crew · role · shift · competency · certification · stakeholder · commitment.

**6 · Space & time.** site · zone · footprint · clearance · clash · window · deadline · holiday · weather.

**7 · Money & agreements.** budget · estimate · purchase-order · invoice · payment · change-order · contract · clause · claim.

**8 · Documents & decisions.** design · spec · procedure · runbook · report · decision · action-item · meeting · lesson · assumption · hypothesis.

**9 · Meta.** tag · view · template · ruleset · event · override · alias.

Each is characterized by the *shape* of the constraints it commonly carries — a `weather` atom's `duration` is a forecast window; a `hazard`'s `duration` is exposure time; a `commitment`'s `duration` is time-to-fulfillment. The engine does not know the difference.

Note the meta family. A **view** is a saved query — and, yes, an atom. A **template** produces instances of atoms on instantiation. A **ruleset** generates constraint atoms from a policy. An **event** is an atom's state transition, itself atomic and audit-worthy. An **alias** is the assertion that two atoms refer to the same real thing (used when two subgraphs merge). An **override** is a time-boxed suspension of a constraint. These are the machinery for describing the graph *in the graph's own language*. See § 12 · round 7.

## 12. What can't be an atom? — a recursive proof-by-contradiction

I built the taxonomy above by asking, for every noun a construction organization uses, "would treating it as an atom break anything?" The answer, so far, is no. That should be suspicious. The model may be too accommodating. So let me try to break it.

The pattern is: state the strongest claim I can that some category is not an atom, and then attempt to disprove it. The disproof either succeeds or the claim stands. I get to round eight and stall. I don't think there's a round nine but I would like to be wrong.

**Round 1. Relationships aren't atoms.** — Disproof: a relationship *is* already an atom in this schema. The two link types (`contains`, `requires`) are relationships and they are first-class. If you want a labeled relationship — "blocks", "duplicates", "co-signed-by" — give it its own atom type, put the two endpoints in its `requires` list, and it now carries its own state, its own narrative, its own audit trail. Every relationship is an atom looking sideways.

**Round 2. A number isn't an atom.** — Disproof: a raw number isn't, but a *measurement* is. A measurement has a source, a time, a unit, an instrument, a tolerance, an observer. Those are fields. The number is a leaf inside a measurement atom. Same for a cost figure (bound to a budget line), a duration (bound to a work atom), a coordinate (bound to a footprint expression). Loose numbers are homeless. Bind them to context and they become atoms.

**Round 3. A moment in time isn't an atom.** — Disproof: a moment is a zero-duration verified atom — it is a milestone. A moment with a name is a milestone with a name. A moment nobody noticed passing is still an atom, just one with an empty narrative. The claim conflates "a moment" with "the number denoting the moment." The number is a field. The moment is the atom.

**Round 4. Trust between two people isn't an atom.** — Disproof: it is a *relationship* atom, and a fragile one. Its kind is manual; its tags include `narrative-required`. It gets updated when the two people work together on other atoms — you write about the work, and the trust atom's narrative accrues history. Its "duration" is the time to rebuild after a breach. Its "requires" is a series of small commitments kept. This is why work relationships are engineerable rather than mystical.

**Round 5. An assumption isn't an atom because it isn't real.** — Disproof: an assumption is an atom of `kind: manual` and `type: assumption`. Its `manualDone` is true until someone challenges it, at which point it flips to false and every atom that requires it becomes not-ready. Assumptions *are* real — they are load-bearing in the graph. The whole schedule is built on them. The engine's job is to know which ones are being trusted and by whom.

**Round 6. The absence of something isn't an atom.** — Disproof: two ways to model absence. First, a missing atom in a `requires` list — the schedule tells you exactly what's absent by what fails to derive. Second, a first-class `void` atom explicitly noting a decision *not* to do something — a de-scoped work package, a rejected change order. Both are atoms. What isn't in the graph is still describable by the graph — the shape of the hole is data.

**Round 7. The ACE engine itself can't be an atom.** — Disproof: the engine is a `ruleset` atom in the meta-graph. Its `derive` function is a `procedure` atom. Its `contains` field points at the type registry. Its `requires` field points at the axioms it depends on — "no cycles," "finite atoms," "acyclic requires." The settings that configure ACE are atoms. The schema is an atom. The engine describes itself in its own language. This is not a coincidence — it is a soundness condition. If the modeling primitive cannot describe itself, it is expressively incomplete for its own domain.

**Round 8. "Not an atom" isn't an atom.** — Disproof: the category "not-an-atom" is a view — a saved query with the predicate `false`. Every view is an atom. So the concept of a thing that isn't an atom is itself an atom pointing at an empty set. The claim eats itself.

**The recursion terminates.** Every candidate for "not an atom" turned out to be an atom describing an atom-shaped thing. This is not a coincidence. It is what it *means* for a modeling primitive to be complete: the graph is closed under its own description. That's the property that lets one engine run the whole show.

I invite round nine. The strongest case I have not adequately addressed is *contradictions* — an atom that requires the negation of another atom. I currently model this as a `constraint` atom whose gate is "predecessor is `manualDone = false`," which works but feels like a fixed point where the fix is a language feature (see § 13.3, "negative links"). This is the first place I would attack.

## 13. Novel ideas and open questions

### 13.1. Fuzzy links (probabilistic requires)

`requires` currently returns Boolean. What if the edge carries a probability? `a requires b with p = 0.8` means "there is an 80% chance a needs b done first." Under Monte Carlo, this becomes a Bernoulli draw per run. The critical path becomes a *probable* path with confidence intervals. This handles cases where the design isn't finalized — you know the pump might need the extended base plate, but you won't be sure until the vibration analysis lands.

The math is well-behaved: a fuzzy DAG is still acyclic; percolation gives you the completion probability of any downstream atom.

### 13.2. Predicate targets

Following § 10, generalize `requires` targets to arbitrary predicates over the atom set. This subsumes named references (`id = 'REBAR'`) and capacity queries (`type = 'crew' and status = 'idle'`). Every `requires` edge is `requires(predicate)`. The engine's readiness check becomes: *does any atom in the current graph satisfy the predicate?* Named references become the degenerate case.

The cost is that derive is no longer O(V + E) in the worst case — a predicate that scans all atoms is O(V). But predicates that index cleanly (`type = 'x'` with a type index) remain fast, and the expressive win is enormous.

### 13.3. Negative links ("excludes")

The one primitive I am tempted to add is a link that says "this atom is incompatible with that atom." Not `requires` — the opposite. `a excludes b` means the graph is over-determined if both are done. This handles interference (§ 6 · clash), mutual-exclusion in resource allocation, and contradictions in claims. The disproof of § 12 · round 8 handled this by delegating to a `constraint` atom, which works but is a workaround.

If a negative link is admitted, the engine must extend the derive step to a soundness check — the graph is over-determined iff any atom pair `(a, b)` with `a excludes b` are both done. This is still one pass. The question is whether it should be a separate link type or a new atom type (`clash`) whose `requires` list is the two mutually-exclusive atoms and whose `manualDone` is the exclusion assertion. I lean toward the latter for now — no new primitive — but I acknowledge this is the boundary of the theory.

### 13.4. Meta-atoms — templates that produce atoms

A **template** is an atom whose evaluation produces other atoms. Instantiating a template of `type = 'iwp-template'` on a specific location generates a work package with a canonical `contains` list (task atoms), `requires` list (predecessor atoms), and initial `duration` triple. This is macro expansion within the graph. The template atom itself is a `contains`-parent of every instance it produces, so the graph records provenance.

The philosophical implication is that the *taxonomy of atom types* (§ 11) is itself an atom — a template that produces the schema each atom type uses. The schema is data.

### 13.5. Views-as-atoms

A view is a saved query. Every view is an atom. This is not a decoration. It means views can `require` other views, form their own dependency graph, and be triaged like anything else. "The Wednesday morning report" is an atom whose `requires` list is the queries it aggregates. If one of those queries stops working (e.g., its predicate references a removed field), the report atom is not-ready and shows up on triage as a broken report.

### 13.6. Time-varying atoms

An atom's state changes over time. A material atom is `not-delivered` until Tuesday, then `on-shelf` until Thursday, then `consumed`. Currently we model this by mutating `manualDone`, which erases history.

An improvement is to represent an atom's state as a *sequence of event atoms* — each state transition is an atom of `type = event`, `contains`ed by the parent atom, ordered by timestamp. The atom's *current* state is the last event's kind. History is queryable; the audit trail is not a log but a first-class subgraph.

This has an ergonomic cost — most atoms don't need this precision, and the graph grows. The right rule is probably: state transitions default to overwriting `manualDone`, and atoms tagged `event-sourced` accrete event atoms instead. Both live in the same graph.

### 13.7. Multi-observer graphs (subjective truth)

Two people looking at the same project disagree about what's done. Currently we model this by making the observer implicit and letting the graph carry an authoritative view. A more honest model is to attach each state transition to an `observer` atom (a person or role) and let the graph carry multiple concurrent beliefs.

Derive over a multi-observer graph asks: *from whose point of view?* This is unavoidably political and I do not know how to make it precise. But the framework survives: the observer is an atom, the belief is an event atom attributed to that observer, and the derived state is a query over an observer.

### 13.8. Federated graphs

A general contractor and a subcontractor each run their own ACE instance. Atoms cross the boundary as `alias` atoms — the GC's `MAT-RBR-014` is aliased to the sub's `PO-2091-item-4`. Alias atoms carry the mapping; derive resolves through them.

This is how ACE handles the situation where one party's graph must reference another party's atom without either party surrendering ownership of the source of truth. The alias atom carries the trust boundary as data.

## 14. Related work and lineage

**Theory of Constraints (Goldratt, 1984).** The intellectual grandparent. The bottleneck is the constraint that governs system throughput. ACE takes the observation seriously enough to make constraints the *primitive* rather than a separate concept overlaid on a scheduling system.

**Entity-Component-Systems** (game engines: Unity ECS, Bevy). The pattern of "one flat record, many typed views on it" is standard in performance-critical game engines. ACE imports this pattern and applies it to project controls. The `type` field is the component tag; the seven fields are the shared component data.

**Datalog and logic programming** (Prolog, Datomic, Datafun). Deriving state from a graph by fixed-point iteration is precisely how a Datalog engine evaluates recursive queries. The `derive` step in § 4 is a monotonic Datalog rule: `done(a) :- forall b in contains(a): done(b), forall c in requires(a): done(c)`. A Datalog engineer will recognize this immediately. What ACE contributes over vanilla Datalog is (a) the two-link discipline and (b) the taxonomy of five kinds.

**Petri nets and workflow modeling.** The relationship between an atom and its `requires` predecessors is the token-passing pattern of a Petri net. Petri nets are more expressive (they carry token counts) but are notoriously unwieldy for humans to author. ACE trades some expressive power for authorability. Where a Petri net is needed, an atom's `requires` predicate can carry the counting logic.

**Advanced Work Packaging (Construction Industry Institute, 2013).** AWP is the current-generation methodology for construction project controls. It decomposes work into CWPs and IWPs and gates work on "five constraints cleared" (design, materials, equipment, personnel, safety). ACE is a data model that AWP could adopt without changing its practice: the five constraints become five `requires` edges from an IWP to five constraint atoms of the appropriate types.

**Primavera P6 and MS Project.** The incumbents in scheduling. Both maintain separate tables for activity, relationship, resource, calendar, and code. Both require a Windows install or a database license. Both are absent from the browser. ACE is a data-model proposal, not a P6 replacement per se; a P6-flavored view of an ACE graph is straightforward. The point is that the P6 data model was not the only option, and it is worth revisiting.

**Datomic (Rich Hickey, 2012).** Datomic is a transaction-log-first, immutable-database model where all state is derived from a log of assertions. ACE shares the philosophy that state should not be stored but derived. Datomic maintains a temporal history; ACE (§ 13.6) contemplates the same via event atoms.

## 15. Adversarial questions I have asked myself

**Does the model scale?** — Derive is O(V + E). Contains is a tree; requires is a DAG. For 10^5 atoms with average degree 3, one derive pass is on the order of milliseconds. Monte Carlo N = 1000 is one derive per run, so ~seconds for 10^5 atoms. Beyond 10^6 atoms, structure-sharing between runs starts to matter and the derive pass should exploit topological order caching. Nothing in the model precludes scaling; the current implementation is not tuned for it.

**What if I need cycles?** — You don't. Every case I have found where "circular dependency" seems to exist is either (a) a temporal illusion (A blocks B at t=1, B blocks A at t=2 — model as separate atoms per time) or (b) a mis-specified requirement (introduce an intermediate `assumption` atom that breaks the cycle explicitly). Rejecting cycles at insert time is a strong invariant.

**Isn't the taxonomy just tags in a costume?** — Somewhat, yes. `type` is a string. What the taxonomy contributes is a *canon* — a shared vocabulary organizations can commit to. Without it, everybody makes up their own strings and interoperability breaks. With it, an alias atom (§ 13.8) can map between organizations that both speak the same canon.

**How is this different from RDF?** — RDF (Resource Description Framework) has a similar spirit — everything is a triple (subject-predicate-object). RDF is expressively too general: it puts no structure on the predicate, so it can encode anything and computes nothing. ACE is more disciplined: two link types, five completion kinds, one derive. The rigidity is the point.

**What about writes? Multi-user?** — The paper deliberately does not address multi-user concurrency. In the demo the graph is in-memory single-user. A production implementation would use event sourcing (§ 13.6) — every mutation is an event atom appended to a log, and the current state is derived from the log. This is well-understood territory; the theory does not depend on the choice.

**Are there projects too small for this?** — Yes. If your project fits in a spreadsheet, use a spreadsheet. ACE's value grows with the number of *cross-cutting concerns* — the moment you have work AND materials AND constraints AND regulatory AND spatial AND financial in one project, the reconciliation cost dominates and one graph pays off.

**Are there projects too large?** — Not that I have found. The Advanced Work Packaging community operates comfortably at 10^5 IWPs on major-project megaprojects. That is well within one browser's memory. Above that, sharding on the `area` boundary is straightforward.

## 16. Implementation notes

The reference implementation is a single ~2,500-line HTML file. The engine core — atom store, add, link, cycle check, derive, CPM walk, Monte Carlo sampler — is ~200 lines of vanilla JavaScript. The rest is the demo shell: rendering, live exhibits, workbench, procedural document, parametric geometry.

No dependencies. No build step. No server. The tool ships in one file that can be opened offline, saved, and edited in a text editor. This is not a design flex; it's a stress test of the claim that the model is simple. If a working prototype needed a framework and a build pipeline, the "one primitive" claim would ring hollow.

The demo carries eight or nine live interactive exhibits — you scroll to derive and click atoms; you scroll to forecast and press Run 1000; you scroll to space and drag the sliders. All exhibits share one ACE instance in memory. Every mutation in any exhibit is visible in every other. This is the strongest evidence I can produce that the single-primitive claim is not just philosophically satisfying but *operationally* real.

## 17. Closing

One primitive. Two link types. One derive. Everything else is a view.

I have been unable to break the claim in eight rounds of trying. The taxonomy exhibits 72 types without a schema change. Space and time are the same primitive as work. Narrative is a first-class atom, forced when the tag demands it. Views, templates, and even the engine itself live in the graph as meta-atoms.

The critical open questions are round nine of § 12 (which I have not found) and the treatment of negative links / contradictions (§ 13.3), which currently reduces to a workaround I am not fully satisfied with. Predicate-target `requires` (§ 13.2) is the largest generalization I would make; fuzzy links (§ 13.1) is the second.

If the theory is right, the entire project-controls industry is running on a stack that is one primitive too many, five separate systems whose reconciliation cost is a full-time job that produces no value. That is a big claim. It should be attacked hard. I welcome any counter-example that stands.
